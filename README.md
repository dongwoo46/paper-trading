실시간 시장 데이터를 수집하고, 사용자가 정의한 매매 규칙을 기반으로 자동 매수·매도를 시뮬레이션하며,  
그 결과를 가상 계좌·포트폴리오·거래 이력·성과 지표로 검증할 수 있는 개인용 모의투자 플랫폼입니다.

이 프로젝트의 핵심 목적은 단순히 "가짜로 사고파는 앱"을 만드는 것이 아니라,  
**내가 생각한 투자 전략이 실제 시장 데이터 위에서 어떻게 동작하는지 검증할 수 있는 실험 환경**을 만드는 데 있습니다.

즉, 이 프로젝트는 다음 3가지를 동시에 해결하려고 합니다.

1. **실시간 시장 데이터 수집**
   - 외부 증권사/거래소 API(예: 한투, 업비트 등)로부터 가격, 지수, 호가 등의 데이터를 실시간으로 수집합니다.
   - 수집된 최신 데이터는 빠른 조회를 위해 Redis에 저장합니다.
   - 장기 분석 및 복기를 위해 히스토리 데이터는 PostgreSQL에 주기적으로 적재합니다.

2. **모의 주문 및 체결 시뮬레이션**
   - 사용자는 가상 계좌를 기반으로 시장가/지정가 주문을 등록할 수 있습니다.
   - 시스템은 실시간 시장 데이터를 기준으로 주문의 체결 여부를 판단하고, 예수금·보유 수량·평균 단가·손익을 갱신합니다.
   - 초기 버전에서는 단순 체결 규칙으로 시작하여, 이후 부분 체결·호가 잔량·슬리피지 등으로 확장할 수 있습니다.

3. **자동매매 전략 검증**
   - 사용자는 특정 조건(예: 가격 돌파, 손절, 익절, 이동평균 교차 등)에 따라 자동 주문을 실행하는 전략을 등록할 수 있습니다.
   - 전략 엔진은 실시간 가격 상태를 기준으로 조건을 평가하고, 조건이 충족되면 모의 주문을 생성합니다.
   - 이를 통해 사용자는 "내 규칙이 실제로 수익이 나는지", "어떤 상황에서 손실이 커지는지"를 검증할 수 있습니다.

## 프로젝트가 해결하려는 문제

실제 투자 전략을 검증하려고 할 때 보통 다음과 같은 문제가 있습니다.

- 단순 차트 관찰만으로는 전략을 객관적으로 검증하기 어렵다.
- 머릿속 규칙은 있어도, 그 규칙을 반복적으로 테스트할 수 있는 도구가 없다.
- 실제 돈을 넣기 전 전략을 충분히 검증할 안전한 환경이 필요하다.
- 실시간 데이터와 주문/체결/손익 흐름을 함께 시뮬레이션할 수 있는 구조가 필요하다.

이 프로젝트는 위 문제를 해결하기 위해 다음과 같은 방향으로 설계되었습니다.

- **최신 시장 상태**는 빠르게 조회할 수 있어야 한다.
- **주문/체결/잔고/포트폴리오 상태**는 일관성 있게 관리되어야 한다.
- **전략 규칙**은 자동으로 평가되고 반복 실행 가능해야 한다.
- **히스토리 데이터**는 나중에 분석 및 복기할 수 있어야 한다.

---

## 프로젝트 목표

이 프로젝트의 1차 목표는 다음과 같습니다.

- 실시간 시장 데이터를 수집할 수 있다.
- 최신 가격 정보를 빠르게 조회할 수 있다.
- 사용자가 가상 계좌 기반으로 주문할 수 있다.
- 주문이 시장 데이터 기준으로 체결된다.
- 체결 결과가 계좌와 포트폴리오에 반영된다.
- 자동매매 전략을 등록하고 실행할 수 있다.
- 전략의 결과를 거래 이력과 손익으로 검증할 수 있다.

즉, 이 프로젝트의 본질은 다음 한 문장으로 정리할 수 있습니다.

> **실시간 시장 데이터 위에서 자동매매 전략을 검증할 수 있는 개인용 모의투자 시뮬레이션 시스템**

## 핵심 기능

### 1. 실시간 시장 데이터 수집

- 외부 API(WebSocket/REST)를 통해 종목/코인/지수 가격을 수집합니다.
- 최신 가격은 Redis에 즉시 반영합니다.
- 시장 데이터 히스토리는 PostgreSQL에 주기적으로 적재합니다.
- 초기에는 "최신가 + 기본 히스토리 저장"에 집중하고, 이후 호가/캔들/지표로 확장할 수 있습니다.

#### 기대 효과

- 주문 처리 시 DB가 아닌 Redis 기반으로 빠르게 최신가를 참조할 수 있습니다.
- 실시간 데이터와 장기 이력 데이터를 분리하여 저장 목적을 명확히 할 수 있습니다.

---

### 2. 가상 계좌 관리

- 사용자는 실제 계좌가 아닌 가상 계좌를 사용합니다.
- 초기 예수금을 기준으로 주문을 수행합니다.
- 체결 결과에 따라 예수금, 보유 종목, 평균 단가, 평가손익, 실현손익을 갱신합니다.

#### 관리 대상

- 예수금
- 보유 종목 수량
- 평균 매입 단가
- 미실현 손익
- 실현 손익

---

### 3. 모의 주문 기능

- 시장가 매수/매도
- 지정가 매수/매도
- 주문 조회
- 주문 상태 변경
- 미체결 주문 관리

#### 초기 체결 규칙

- 시장가 주문은 현재가 또는 1호가 기준 즉시 체결
- 지정가 주문은 조건 충족 시 체결, 미충족 시 대기
- 초기에는 전량 체결 가정
- 추후 부분 체결, 슬리피지, 호가 잔량 반영 확장 가능

#### 왜 이렇게 시작하는가

초기 버전의 핵심은 "거래소 수준 매칭 엔진"이 아니라  
**주문 → 체결 → 계좌 반영 → 손익 계산** 흐름을 정확하게 만드는 것입니다.  
따라서 처음부터 복잡한 주문장 시뮬레이션보다는 단순 체결 규칙으로 핵심 도메인을 안정화하는 것이 우선입니다.

---

### 4. 포트폴리오 관리

- 현재 보유 종목 목록 조회
- 종목별 평가금액/평가손익/수익률 계산
- 총 자산 및 총 수익률 계산
- 거래 이력 기반 성과 확인

#### 목적

사용자가 현재 전략으로 인해 어떤 포지션을 보유하고 있으며,  
실시간 가격 기준으로 수익과 손실이 어떻게 변하는지를 직관적으로 확인할 수 있어야 합니다.

---

### 5. 자동매매 전략 등록 및 실행

사용자는 특정 규칙을 전략으로 등록할 수 있습니다.

예:

- 특정 가격 이상 돌파 시 매수
- 특정 가격 이하 하락 시 손절
- 수익률 5% 도달 시 익절
- 손실률 -3% 도달 시 매도
- 이동평균 교차 시 진입/청산

#### 전략 엔진 역할

- 실시간 최신 가격 조회
- 전략 조건 평가
- 조건 충족 시 자동 주문 생성
- 생성된 주문을 모의 체결 엔진에 전달

#### 핵심 목적

이 프로젝트는 단순 거래 기록 앱이 아니라  
**사용자가 정의한 규칙이 실제로 유효한 전략인지 검증하는 시스템**이기 때문에,  
전략 등록 및 자동 실행 기능이 핵심 축 중 하나입니다.

---

### 6. 거래 이력 및 성과 분석

- 주문 이력 조회
- 체결 이력 조회
- 종목별 손익 조회
- 전략별 성과 비교
- 전체 수익률 추적

#### 향후 확장 가능한 지표

- 승률
- 평균 손익비
- 최대 낙폭(MDD)
- 전략별 누적 수익률
- 구간별 성과 비교

## 시스템 관점에서의 동작 방식

이 시스템은 크게 두 가지 축으로 동작합니다.

### A. 시장데이터 수집 축

외부 API로부터 실시간 데이터를 수집하고, 이를 내부 시스템이 빠르게 활용할 수 있도록 가공/저장합니다.

#### 흐름

1. 외부 거래소/증권사 WebSocket 연결
2. 실시간 가격/지수/호가 수신
3. 최신 상태를 Redis에 반영
4. 히스토리 저장용 이벤트를 버퍼에 적재
5. 일정 시간/건수 기준으로 PostgreSQL에 batch insert

#### 분리 이유

최신 상태 조회와 장기 이력 저장은 사용 목적이 다릅니다.

- Redis: 현재 상태 조회
- PostgreSQL: 이력 분석 및 복기

이 둘을 분리하면 실시간성과 저장 효율성을 동시에 확보할 수 있습니다.

---

### B. 거래/전략 실행 축

사용자 주문과 자동매매 전략을 실행하고, 그 결과를 가상 계좌 상태에 반영합니다.

#### 흐름

1. 사용자가 주문을 등록하거나 전략이 자동 주문을 생성
2. 시스템이 Redis에서 최신 시장 상태 조회
3. 체결 가능 여부 판단
4. 체결 시 주문 상태 변경
5. 계좌/포트폴리오/손익 반영
6. 거래 이력 저장

#### 분리 이유

시장데이터 수집과 주문 도메인은 관심사가 완전히 다릅니다.

- 시장데이터 수집: 외부 스트림 ingest
- 거래 처리: 내부 비즈니스 정합성

따라서 런타임 특성과 책임이 다른 두 영역을 분리하는 것이 유지보수성과 확장성 측면에서 유리합니다.

## 아키텍처 개요

### `market-collector`

실시간 시장데이터를 수집하는 서비스입니다.

#### 주요 책임

- 외부 WebSocket/REST 연동
- 실시간 가격/지수 데이터 수집
- Redis 최신가 반영
- 히스토리 데이터 batch 적재

#### 기술 선택 이유

이 서비스는 수많은 실시간 이벤트를 비동기적으로 처리해야 하므로,  
스트리밍·연결 유지·재연결·backpressure 처리에 적합한 WebFlux 기반으로 설계합니다.

---

### `trading-api`

모의주문, 체결, 포트폴리오, 전략 실행을 담당하는 서비스입니다.

#### 주요 책임

- 가상 계좌 관리
- 주문 생성/조회
- 체결 처리
- 포트폴리오 조회
- 전략 등록/수정/실행
- 손익 계산

#### 기술 선택 이유

이 서비스는 주문/체결/잔고/포트폴리오 같은 강한 정합성이 필요한 비즈니스 도메인을 다룹니다.  
따라서 트랜잭션, JPA, 도메인 모델링이 자연스러운 Spring MVC 기반으로 설계합니다.

---

### `trading-web`

사용자가 시스템을 직접 사용하는 웹 애플리케이션입니다.

#### 주요 역할

- 실시간 가격/포트폴리오 조회
- 주문 입력
- 전략 등록
- 거래 이력 및 성과 조회

#### 기술 선택 이유

빠른 개발 속도와 UI 구성의 유연성을 고려해 React 기반으로 설계합니다.

# 추가 기능 AI활용 모의투자 전략 연구 시스템

## 목표

외부 시장 데이터, 뉴스, 공시, 금리, 재무 정보, 과거 전략/백테스트 결과를 종합해  
AI가 **전략 후보를 제안**하고, 이를 **백테스트와 사람이 검토**하여 실제 적용 여부를 결정하는 시스템을 구축한다.

---

## 핵심 원칙

- AI는 **주문을 직접 실행하지 않는다**
- AI는 **전략 후보/전략 추천**만 생성한다
- 실제 적용 여부는 **사용자 승인** 또는 제한된 운영 정책에 따라 결정한다
- 승인된 전략만 실시간 운영 환경에 반영한다
- 전략 생성, 수정, 평가, 승인 이력을 모두 버전으로 관리한다

---

## 주요 기능

### 1. 시장 데이터 수집

- 거래소 / 증권사 API 및 WebSocket을 통한 시세 데이터 수집
- 뉴스, 공시, 금리, 경제지표, 재무 데이터 수집
- 실시간 데이터와 히스토리 데이터를 분리 저장

### 2. 사건 분석 및 시뮬레이션

- 뉴스/공시/매크로 이벤트를 분석해 사건 유형 분류
- 특정 사건이 시장, 섹터, 종목에 미칠 영향을 시뮬레이션
- MiroFish + 로컬 LLM을 활용해 다중 시나리오 생성
- 강세 / 중립 / 약세 시나리오와 예상 영향 기간 도출

### 3. RAG 기반 전략 아이디어 생성

- 전략 관련 문서, 과거 성과, 실적, 뉴스, 금리, 공시, 시장 레짐 정보를 검색
- 검색된 근거를 바탕으로 신규 전략 후보 생성
- 전략 제안 시 반드시 근거 데이터와 설명을 함께 생성

### 4. 전략 DSL 변환

- 자연어 전략 설명을 구조화된 전략 DSL(JSON)로 변환
- 진입 조건 / 청산 조건 / 포지션 크기 / 보유 기간 / 리스크 제한 명시
- 사람이 읽을 수 있고, 백테스트 엔진이 실행 가능한 형태로 관리

### 5. 백테스트 및 검증

- 생성된 전략 후보를 과거 데이터 기준으로 백테스트
- 수익률, MDD, Sharpe, 승률, 손익비, turnover 평가
- baseline 전략과 비교 검증
- 검증 실패 전략은 자동 수정 루프로 전달

### 6. 전략 자동 수정 루프

- LangGraph 기반으로 전략 생성 → 평가 → 수정 → 재검증 반복
- 파라미터 조정, 룰 보강, 필터 조건 추가/삭제 수행
- 무한 수정 방지를 위해 수정 횟수, 변경 범위, 평가 기준을 제한
- 모든 수정 내역과 수정 이유 기록

### 7. 사람 승인(Human-in-the-loop)

- 전략 후보, 근거 데이터, 백테스트 결과, 리스크 요약 제공
- 사용자가 전략 적용 여부를 직접 결정
- 승인된 전략만 운영 가능한 상태로 승격

### 8. 실시간 전략 운영

- 승인된 전략만 실시간 시세 스트림에 적용
- Python 전략 엔진이 실시간 signal 생성
- Spring이 signal을 받아 주문 가능 여부, 리스크, 중복 주문을 검증
- 검증 통과 시에만 모의 주문 생성 및 포트폴리오 반영

### 9. 전략 버전 및 실험 관리

- 전략 family / strategy version / parent version 관리
- 생성 주체(LLM/사용자), 수정 이유, 승인 상태, 성능 이력 저장
- 어떤 전략이 왜 수정되었는지 추적 가능하게 유지

- 프롬프트 템플릿
- structured output 생성
- 툴 호출

### LangGraph

- 생성 → 백테스트 → 평가 → 수정 → 승인 루프 구성
- 장기 실행, 재시도, 사람 승인 지점 관리
- 전략 연구 파이프라인 오케스트레이션

## RAG 확장 방향

### 기본 RAG

- 뉴스
- 공시
- 금리
- 재무제표
- 기존 전략 문서
- 백테스트 결과
- 시장 레짐 분석 결과

### 확장 RAG

- 기업 ↔ 섹터 ↔ 지표 ↔ 사건 ↔ 정책 ↔ 원자재 관계 연결
- 과거 유사 사건과 유사 전략 검색
- multi-hop retrieval 기반 아이디어 생성

## Neo4j 활용 후보

### 적합한 경우

- 기업/산업/정책/거시경제/공급망 관계를 따라가며 전략 아이디어를 뽑고 싶을 때
- 단순 문서 검색보다 관계 기반 탐색이 중요할 때
- “이 사건이 어떤 섹터와 종목에 어떻게 연결되는가?”를 설명 가능하게 만들고 싶을 때

### 과한 경우

- 단순 뉴스 요약과 벡터 검색 정도만 필요할 때
- MVP 단계에서 관계 그래프를 아직 충분히 정의하지 못했을 때
