## 이 프로젝트란?

**개인용 모의투자 시뮬레이션 플랫폼**이다.

실시간 시장 데이터를 수집하고, 사용자가 정의한 매매 전략을 기반으로 자동 매수·매도를 시뮬레이션하며,
그 결과를 가상 계좌·포트폴리오·거래 이력·성과 지표로 검증할 수 있는 시스템이다.

### 핵심 목적

> 내가 생각한 투자 전략이 실제 시장 데이터 위에서 어떻게 동작하는지 검증할 수 있는 실험 환경

### 지원 거래 모드

| 모드 | 설명 |
|------|------|
| 로컬 모의투자 | 로컬 원장 기반 자체 모의투자 |
| 연동 모의투자 | 한국투자증권(KIS) 모의투자 계좌 연동 |
| 실시간 거래 | 실제 증권사/거래소 계좌 연동 실거래 |

### 서비스 구성

| 서비스 | 언어/기술 | 역할 |
|--------|----------|------|
| `trading-api` | Java 21 / Spring MVC | 주문·체결·계좌·포트폴리오·전략 실행 |
| `collector-api` | Kotlin / Spring MVC+WebFlux | 실시간 시세 수집, Redis 반영, 히스토리 적재 |
| `collector-worker` | Python | 일별 시세 수집 워커 (yfinance, pykrx 등) |
| `trading-web` | React / TypeScript | 사용자 UI |

## 개발 원칙

### 1. 클린 아키텍처 (Clean Architecture)

- 레이어 의존 방향: `Controller → UseCase(Service) → Domain ← Infrastructure`
- 도메인 레이어는 Spring, JPA 등 외부 프레임워크에 의존하지 않는다.
- 인프라(DB, Redis, 외부 API)는 Port/Adapter 패턴으로 교체 가능하게 구성한다.
- **Java (trading-api) 패키지 구조:**
  ```
  com.papertrading.api
  ├── domain/         # 순수 도메인 모델, 비즈니스 규칙
  ├── application/    # UseCase, 응용 서비스 (오케스트레이션)
  ├── infrastructure/ # JPA 구현체, Redis 클라이언트, 외부 API 어댑터
  └── interfaces/     # REST Controller, DTO, Request/Response
  ```

### 2. 단일 책임 원칙 (SRP)

- 클래스/함수는 하나의 이유로만 변경되어야 한다.
- Service는 하나의 Use Case만 담당한다.
- Entity는 상태와 도메인 규칙만 갖는다. DB 쿼리, HTTP 호출은 절대 넣지 않는다.

### 3. 기획 → 설계 → 개발 → 테스트 프로세스

```
기획 (요구사항 정의) → 설계 (ERD, API, 아키텍처) → 개발 (구현) → 테스트 (단위/통합)
```

- **기획과 설계 단계는 반드시 사용자의 허가를 받은 후 다음 단계로 넘어간다.**
- 사용자가 명시적으로 승인하면 AI가 기획·설계를 자율적으로 진행할 수 있다.
- 개발 중 기획이 바뀌면 반드시 사용자와 합의 후 진행한다.

## .agents 폴더 구조 (서비스 내부)

각 서비스 내부에 `.agents/` 폴더를 두고 아래 구조로 관리한다.

```
.agents/
├── README.md          # 개발 이정표 — 현재 진행 상황 및 다음 작업
├── feature/           # 완료된 기능 설명 (기능명.md)
│   └── README.md      # API 인덱스 + 웹 연결 상태 (항상 최신 유지)
└── rule/              # 특수 버그/장애 발생 시 재발 방지 기록
```

- **feature/**: 기능 완료 시마다 `{기능명}.md` 생성, 목적·변경 API/DB/엔티티·검증 결과·후속 TODO 포함
- **feature/README.md**: API 인덱스 + 웹 연결 상태 (항상 최신 상태만 유지, 과거 상태 남기지 않음)
- **rule/**: 특수 버그·장애 발생 시 증상·재현 조건·원인·해결·재발 방지 체크리스트 기록
- **README.md**: 현재 개발 진척도와 다음 작업 목록을 이정표 형식으로 관리

## 세션 컨텍스트 경고

AI는 컨텍스트 사용량을 정밀하게 측정할 수 없다. 대신 **대화 길이, 읽은 파일 수, 도구 호출 횟수**를 기반으로 경고를 판단한다.

### 경고 기준

시스템이 이전 메시지를 자동 압축하기 **전에** 경고를 줘야 한다. 아래 상황 중 하나라도 해당되면 경고를 표시한다:

- 대화가 매우 길어졌고, 다수의 파일을 읽고 수정한 경우
- 도구 호출이 수십 회 이상 누적된 경우
- 응답 생성이 느려지거나 이전 내용을 잘 기억 못 하는 징후가 보이는 경우

### 경고 타이밍

**성능 저하나 자동 압축이 발생하기 전에** 반드시 경고를 준다.
압축이 발생하면 이전 대화 맥락이 손실되어 작업 연속성이 끊긴다.

### 경고 문구

```
⚠️ 세션 컨텍스트가 많이 소모되었습니다. 자동 압축 전에 새 세션으로 전환하는 것을 권장합니다.
```

## 공통 버그 방지 규칙

- 금액·수량 계산에는 `BigDecimal`을 사용한다. `double`, `float` 절대 사용 금지.
- 트랜잭션 경계는 Application(UseCase) 레이어에서 관리한다.
- N+1 쿼리 방지: 연관 엔티티 조회 시 fetch join 또는 별도 쿼리로 처리한다.
- 주문 상태 변경은 낙관적 락(Optimistic Lock) 또는 비관적 락으로 동시성 보장한다.
- Redis 캐시와 DB 데이터 정합성 불일치 가능 흐름은 명시적으로 설계한다.
- 외부 API(KIS, Upbit 등) 호출은 Circuit Breaker 또는 타임아웃을 반드시 설정한다.
- DTO와 Entity를 혼용하지 않는다. 레이어 경계에서 반드시 변환한다.
- 시크릿/자격증명 원문은 절대 로그에 남기지 않는다.
