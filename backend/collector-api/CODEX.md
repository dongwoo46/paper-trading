# CODEX 공통 규칙

이 문서는 `collector-api` 작업 시 항상 먼저 읽고 지켜야 하는 공통 규칙이다.

## 1) 시작 규칙
- 코딩 시작 전에 반드시 `CODEX.md`, `.codex/README.md`, `.codex/feature/README.md`를 읽는다.
- 작업 범위(기능/수정 파일/검증 방법)를 먼저 확정하고 진행한다.

## 2) 개발/검증 규칙
- 기능 개발 후 반드시 컴파일 또는 빌드를 실행한다.
- 실패 시 원인 분석 후 수정하고, 성공할 때까지 반복한다.
- 백엔드 기본 검증: `.\gradlew.bat compileKotlin`
- 프론트엔드 기본 검증: `npm run build` (`frontend/trading-web`)

## 3) 기능 보고서 규칙
- 기능 완료 시 `.codex/feature`에 기능 보고서를 추가한다.
- 보고서에는 목적, 변경 API/DB/엔티티, 검증 결과, 후속 TODO를 포함한다.

## 4) 문제 재발 방지 규칙
- 개발 중 발견한 버그/운영 이슈는 `.codex/rules`에 재발 방지 규칙으로 기록한다.
- 기록 항목: 증상, 재현 조건, 원인, 해결, 재발 방지 체크리스트

## 5) API 기록 규칙 (중요)
- API를 새로 만들거나 수정하면 반드시 `.codex/feature/README.md`를 즉시 갱신한다.
- 프론트 웹에서 API를 연결/수정한 경우도 반드시 같은 문서에 반영한다.
- 문서에는 최소한 아래 두 가지를 항상 최신으로 유지한다.
  - 백엔드 API 목록
  - 웹 연결 상태(연결 컴포넌트/미연결 사유)

## 6) 최신성 규칙 (중요)
- `.codex/feature/README.md`는 항상 최신 데이터만 유지한다.
- 오래된 상태, 중복 목록, 과거 기준 설명은 남기지 않는다.
