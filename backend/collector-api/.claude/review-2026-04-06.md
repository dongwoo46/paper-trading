## 리팩터링 필요 항목

### 🔴 High — 버그 가능성 / 설계 결함

#### 1. `KisRestPollingScheduler` 이중 기동 위험
- **위치**: `KisRestPollingScheduler.kt:22-26`
- **문제**: `@EventListener(ApplicationReadyEvent::class) pollOnStartup()`과 `@Scheduled(initialDelay=0)`이 둘 다 존재. 기본 설정(`restPollingInitialDelayMillis=0`)이면 두 개가 거의 동시에 `poll()`을 호출. `AtomicBoolean` 가드가 한 번은 막아주지만, 의도가 불명확하고 `@EventListener`가 남아있는 이유가 없다.
- **권장**: `pollOnStartup()` 제거. `@Scheduled`의 `initialDelay=0`이 기동 직후 실행을 커버함.

#### 2. `KisWebSocketCollector.start()` — REST 캐시 초기화 fire-and-forget
- **위치**: `KisWebSocketCollector.kt:64-67`
- **문제**: `restWatchlistService.initCache(mode)`는 `subscribeOn + subscribe()`로 fire-and-forget이고, `wsSubscriptionService.initCache(mode)`는 `startMode()` 내부에서 `.then()` 체인으로 awaited. 두 캐시 초기화의 완료 순서가 보장되지 않는다.
- **권장**: 둘 다 `startMode()` 내에서 `Mono.zip()`으로 병렬 초기화 후 `.then(connectOnce(mode))` 체인.

### 🟡 Medium — 코드 품질 / 유지보수성

#### 3. `KisRestWatchlistService.addSymbol` / `removeSymbol` — Boolean 반환
- **위치**: `KisRestWatchlistService.kt:36-59`
- **문제**: `addSymbol`은 `Boolean`, `KisWsSubscriptionService.addSymbol`은 `SubscriptionChangeStatus`를 반환. 같은 패턴에 일관성이 없다. Boolean은 실패 이유를 알 수 없어 `KisRestWatchlistController`에서 적절한 응답을 구성하기 어렵다.
- **권장**: `SubscriptionChangeStatus`로 통일. `KisRestWatchlistController`는 현재 없으니 먼저 `KisRestWatchlistService`부터 수정.

#### 4. DTO snake_case 필드명 직접 사용
- **위치**: `KisAccessTokenClient.kt:69-73`, `KisApprovalClient.kt:37-41`
- **문제**: `TokenRequest(grant_type, appkey, appsecret)`, `ApprovalResponse(approval_key)` 등 클래스 필드가 snake_case. Kotlin 컨벤션 위반.
- **권장**: camelCase + `@JsonProperty("grant_type")` 방식으로 변경.

#### 5. `KisProperties` — mode 분기 반복
- **위치**: `KisProperties.kt:43-65`
- **문제**: `if (mode == "live") ... else ...` 패턴이 `appKeyFor`, `appSecretFor`, `approvalUrlFor`, `tokenUrlFor`, `restBaseUrlFor`, `websocketUrlFor` 6개 함수에 반복.
- **권장**: 내부 sealed class 또는 data class `ModeConfig(appKey, appSecret, tokenUrl, approvalUrl, websocketUrl, restBaseUrl)`를 두고 `configFor(mode): ModeConfig`를 만들어 6개 함수를 위임.

#### 6. `KisRateLimiter.acquireRest` — magic string
- **위치**: `KisRateLimiter.kt:21`
- **문제**: `if (mode == "live")` 매직 스트링. `KisProperties`와 동일한 패턴이므로 상수 또는 공통 모드 판별 로직으로 추출 필요.

#### 7. 로거 혼용
- **문제**: 일부 파일은 `LoggerFactory.getLogger(javaClass)`, 일부는 `mu.KotlinLogging.logger {}`. 프로젝트 전체에서 하나로 통일 필요.
- **현황**: `KisAccessTokenFileStore`, `KisRestPollingScheduler` → SLF4J 직접 / `KisWebSocketCollector`, `GlobalExceptionHandler` → KotlinLogging

- `KisWsSubscriptionService.addSymbol` — TOCTOU 경쟁 조건을 `DataIntegrityViolationException` 캐치로 방어.
- `GlobalExceptionHandler` — Bad Request / 500 분리 처리 구조 적절.
- `KisWebSocketCollector` — `Retry.backoff` + `doBeforeRetry` 로깅으로 재연결 로직 완성도 높음.
